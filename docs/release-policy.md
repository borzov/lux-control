# Release Policy

LuxControl releases are shipped from GitHub Actions only. Local builds are useful for development checks, but they are not release artifacts.

Repository: `https://github.com/borzov/lux-control`

## Versioning

- Use semantic versions: `MAJOR.MINOR.PATCH`.
- Tag releases as `vMAJOR.MINOR.PATCH`.
- Keep `CHANGELOG.md` updated before creating a release tag.
- The current app bundle version is derived from the tag by `.github/workflows/release.yml`.

## Release Flow

1. Merge release-ready changes to `main`.
2. Ensure `CHANGELOG.md` contains an entry for the release version.
3. Push a signed or normal git tag named `vMAJOR.MINOR.PATCH`.
4. Let GitHub Actions run the `Release` workflow.
5. Publish only the `.zip` and `.sha256` generated by that workflow.

The release workflow runs the test suite, builds the SwiftPM executable in release configuration, packages `LuxControl.app`, uploads the build artifact, and attaches the same files to the GitHub Release.

Release candidates must pass the compatibility matrix for the three latest macOS generations available on GitHub-hosted runners: macOS 26 Tahoe, macOS 15 Sequoia, and macOS 14 Sonoma.

## Signing

The initial public release is unsigned because no Developer ID certificate is stored in GitHub. Users may see the standard macOS warning for apps downloaded outside the App Store. Future signed releases should add notarization to the same GitHub Actions release workflow instead of moving release builds back to a local machine.
